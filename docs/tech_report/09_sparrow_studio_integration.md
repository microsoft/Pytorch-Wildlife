# Sparrow Studio Integration

Sparrow Studio is Microsoft AI for Good Lab's media processing and review application for AI-for-biodiversity workloads. It ships in two flavors, both consuming sparrow-engine as their inference backend.

- **Sparrow Studio Local** — desktop application. Today: Windows (C# / .NET 9.0 / WPF). In progress: macOS and Linux support via Avalonia UI port on .NET 10+. Consumes sparrow-engine as a native dynamic library via C# P/Invoke. Cross-platform story is intentional — Sparrow Studio Local is not Windows-specific by design.
- **Sparrow Studio Web** — server deployments, consumes sparrow-engine via Docker HTTP service.

## Sparrow Studio — what it does

Media processing and review for:

- Camera trap images
- Drone images
- Drone video
- Audio recordings
- Real-time camera trap feeds delivered through email-enabled cameras

Workflow (per `docs/review/sparrow-studio/sparrow_studio_user_manual.md`):

- Batch processing on media
- Selectable AI pipelines
- Review and correct detections
- Annotate outputs
- Manage labels / classes
- Export results (MegaDet v1.5 JSON, COCO JSON, CSV)

The two Sparrow Studio flavors target different deployments:

| Flavor | User | Deployment | Why this form |
|--------|------|------------|---------------|
| Sparrow Studio Local | Desktop users (Windows today; macOS and Linux via Avalonia port) | Native installer per OS, bundles libsparrow_engine as native dynamic library | Native app delivers a single installable without Python runtime dependencies; suits airgapped and regulated environments; Avalonia gives cross-platform UI on .NET 10+. |
| Sparrow Studio Web | Server operators running bulk campaigns | Docker Compose: `sparrow-engine-server` + workers (persistent storage in the `sparrow-data` sibling per Phase 3.7 Track A, deferred construction) | Horizontal scalability; multi-user; stateless; containerized. |

## Sparrow Studio Local — native library integration

**Stack today.** C# / .NET 9.0 / WPF, win-x64. Migrated to sparrow-engine during Phase 1 handoff.
**Stack in progress.** C# / .NET 10+ / Avalonia UI for cross-platform desktop (Windows, macOS, Linux) — same C# P/Invoke call sites, different UI framework.

**Migration.** Sparrow Studio Local was originally integrated against `Microsoft.ML.OnnxRuntime` NuGet — directly calling ORT from C# with preprocessing reimplemented in C#. The sparrow-engine migration moves preprocessing, postprocessing, and model management into libsparrow_engine. Windows integration report: `docs/review/historical/closure/implementation/windows/implementation_report.md` (2026-04-11).

**What the migration moved.** Preprocessing, postprocessing, and model management moved from C# into libsparrow_engine. Sparrow Studio Local now calls libsparrow_engine for:

- Model loading and session caching.
- Image decode, letterbox, normalize (all 5 model types).
- ORT inference (all 5 model types work on both CPU and GPU with automatic fallback).
- Postprocessing (threshold, softmax, NMS already in-graph).
- Export formats (MegaDet v1.5 JSON, COCO JSON, CSV).

Result: a single Rust implementation of pre/post handles both Sparrow Studio flavors. Correctness fixes land once, not in three places (see § 01 — "Codebase divergence across consumers").

**Binding technology.**

- Native library — produced with `cargo build -p libsparrow_engine --features ffi --release --crate-type cdylib` on the target platform. Output name depends on the OS: `sparrow-engine.dll` (Windows), `libsparrow_engine.dylib` (macOS), `libsparrow_engine.so` (Linux).
- `sparrow_engine.h` — generated by `cbindgen` from Rust source at build time. OS-agnostic.
- C# bindings — generated by `csbindgen` into a ready-to-consume static class with `[DllImport("sparrow_engine")]` declarations. The `DllImport` name resolution picks the platform-appropriate library via .NET's standard lookup rules.
- NuGet package — wraps the native library (one binary per target runtime identifier: `win-x64`, `osx-x64`, `osx-arm64`, `linux-x64`, `linux-arm64`) + C# bindings + type definitions into a single Sparrow Studio Local dependency.

Sparrow Studio Local code then calls:

```csharp
// Illustrative — actual API follows csbindgen generation.
var engine = SparrowEngineNative.sparrow_engine_engine_new(config);
var model  = SparrowEngineNative.sparrow_engine_engine_load_model(engine, "mdv6");
var detections = SparrowEngineNative.sparrow_engine_detect(engine, model, imgBytes, opts);
// ... consume detections
SparrowEngineNative.sparrow_engine_model_free(model);
SparrowEngineNative.sparrow_engine_engine_free(engine);
```

**Tested configurations.** All Sparrow Studio Local production model types verified on Windows x64 (2026-04-11):

- MegaDetector v6 and v5a (detection)
- DeepFaune / DFNE (detection)
- HerdNet (tiled detection)
- SpeciesNet Crop (classification)
- AI4G classifier (classification)
- MD_AudioBirds_V1 (audio detection)

CUDA GPU inference is functional with automatic CPU fallback when no GPU is present. Source: `docs/review/historical/closure/implementation/windows/implementation_report.md § 1`. macOS and Linux testing lands with the Avalonia port.

**FFI safety at the Sparrow Studio Local boundary.** C# P/Invoke consumers hold opaque pointers across managed / unmanaged boundaries. libsparrow_engine's `Weak` + `AtomicBool` + `Arc` pattern (see § 05) closes use-after-free and use-after-unload. Sparrow Studio Local can drop the engine at any time; subsequent inference calls on held model handles return an error code instead of dereferencing freed memory.

**Cross-platform considerations.**

- Path handling: Windows paths use `\`, POSIX paths use `/` — sparrow-engine normalizes via `std::path::Path` on parsing. Source code is path-separator-agnostic.
- ORT runtime linking: the static-linked ORT in libsparrow_engine is the shipping approach on all platforms; dynamic loading is dev / test only.
- Code signing: Sparrow Studio Local consumers expect signed binaries. Release builds will be code-signed per platform (Authenticode on Windows, codesign on macOS, GPG or similar on Linux).
- CPU feature detection: libsparrow_engine uses ORT's built-in CPU EP without architecture-specific builds for portability. Newer CPUs with AVX-512 or ARM NEON use it if ORT detects support; older CPUs fall back to baseline instructions.
- ARM support: macOS `arm64` (Apple Silicon) and Linux `arm64` require a native build per target. Rust cross-compilation via cargo handles it when the Rust toolchain is installed for the target. NuGet packages ship per-RID binaries.

## Sparrow Studio Web — HTTP service integration

**Stack.** Docker Compose orchestration. Services: `sparrow-engine-server` (inference) + worker processes. Persistent storage (media files, annotations, query logs) lives in the `sparrow-data` sibling repo per Phase 3.7 Track A (deferred construction); sparrow-engine-server itself stays stateless.

**Migration.** Sparrow Studio Web was originally integrated against NVIDIA Triton Inference Server. Workers spoke Triton's gRPC protocol. Migrated to sparrow-engine-server + sparrow-engine-client during Phase 2.

**Three worker types.** Sparrow Studio Web ships three worker implementations so teams can pick the stack they already know. All three speak HTTP to `sparrow-engine-server` — none link libsparrow_engine directly.

| Worker | Language | Size | Dependency | Use case |
|--------|----------|------|------------|----------|
| `worker_rust` | Rust (reqwest) | ~230 LOC, 15 MB image | sparrow-engine-server HTTP API | Default. Fastest on GPU workloads. |
| `worker_python` | Python (httpx) | ? LOC, ~200 MB image | sparrow-engine-client | Teams on Python infrastructure |
| `worker_local` | Python (tritonclient) | — | Triton gRPC | Legacy — kept during cutover; not a sparrow-engine consumer. |

The 15 ms/image difference between `worker_rust` and `worker_python` (GPU) is HTTP client overhead. On CPU workloads (~2 s/image) the two are indistinguishable. See § 07.

**Python SDK: `sparrow-engine-client`.**

- Pure Python, ~385 LOC. 21 tests.
- Depends only on `httpx>=0.27`.
- Dataclass types match the HTTP response schema 1:1: `ModelInfo`, `Detection`, `BBox`, `Classification`, `AudioSegment`.
- Sync and async APIs. All 5 inference endpoints wrapped.
- `worker_python` uses `sparrow-engine-client` as its only sparrow-engine-side dependency.

**Phase 3 wire-compat event (SRV1).** R1 of Phase 3 final audit-fix extended `ModelResponse` with 5 new fields (`version`, `description`, `onnx_sha256`, `onnx_size_bytes`, `default`). R1 tests for sparrow-engine-client used stale fixtures that didn't exercise the `default: bool` field. When the test fixture didn't set `default`, the client's `ModelInfo(**m)` dataclass unpacked successfully. But any Phase-3 server (where `default` is always serialized) would cause a `TypeError` on unpack.

R2 inquisitor's independent scan caught the break. Fix (commit `4f272a8`): extend `sparrow-engine-client`'s `ModelInfo` dataclass with all 5 new fields (defaulted), make `load_model` symmetric, add backward-compat regression test that exercises both legacy and new wire shapes.

Lesson: **HTTP schema changes are load-bearing for downstream consumers even when the change is additive.** sparrow-engine's audit-fix process now includes a specific "any ModelResponse field change → run sparrow-engine-client test suite" check.

**Handoff package for Sparrow Studio Web.** Phase 2 delivered a self-contained handoff package: docker-compose snippets, worker reference implementations, sparrow-engine-client package, README with migration instructions. Source: `docs/master_plan.md § Phase 2`.

## Interface ownership in practice

Principle from v4: **sparrow-engine defines the interface. Sparrow Studio adapts to it.**

### Example 1: MT-11 extended fields

Sparrow Studio Local CLI needed to display `version`, `description`, `onnx_sha256`, `onnx_size_bytes`, `default` in its model list UI. The fix was not "have Sparrow Studio Local maintain a separate enrichment step." The fix was to extend sparrow-engine's `ModelInfo` type, propagate the fields through FFI / HTTP / Python, and have Sparrow Studio Local read them directly.

Audit-fix R1 caught that the HTTP `ModelResponse` struct hadn't been extended even after the CLI and Python had been. See SRV1 finding above.

### Example 2: Consumer audit rounds

Phase 2.5 consumer audit had 4 rounds specifically because feature rollout across CLI + Python + FFI + HTTP needs to be coordinated. An audit round per consumer:

- R1: found the first round of inconsistencies (9 findings across the 4 consumers).
- R2, R3: fix cycles.
- R4: converged — all 4 consumer surfaces expose the same feature set identically.

This cycle is expensive but the alternative (letting consumer surfaces drift) produces exactly the kind of silent wire breaks Phase 3 audit-fix R2 caught (see SRV1).

## External / independent-release story

spe is intended to release as an independent open-source repository separate from Sparrow Studio. External users can:

- Use the `spe` CLI from GitHub Releases (no Sparrow Studio dependency).
- `pip install sparrow-engine-python` (no Sparrow Studio dependency).
- Run sparrow-engine-server in Docker without any Sparrow-side software (sparrow-engine-server is a generic HTTP inference service).
- Consume libsparrow_engine from their own C / C++ / C# application using the same DLL shape Sparrow Studio Local uses.

The model-agnostic design (§ 03 interface ownership) supports this: new models can be onboarded by a user adding a manifest + ONNX file to their `{model_dir}/`, no libsparrow_engine code change required.
