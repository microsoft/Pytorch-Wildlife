### Running the full suite

```bash
cd sparrow-engine
bash ./scripts/test.sh                           # libsparrow_engine serial, all crates
bash ./scripts/test.sh -p libsparrow_engine --lib         # libsparrow_engine only
bash ./scripts/test.sh -p sparrow-engine-python --no-default-features   # sparrow-engine-python
cd sparrow-engine/sparrow-engine-client && uv run --extra dev pytest -q          # sparrow-engine-client
```

`scripts/test.sh` sources `scripts/ort-env.sh` which sets `LD_LIBRARY_PATH` to find the dynamically-linked ORT. Without it, tests fail to link because the ORT static lib requires `glibc 2.38+` (dev machine has 2.35).

### Test isolation for the engine singleton

All 18 `engine::tests` in `sparrow-engine/libsparrow_engine/src/engine.rs` are tagged `#[serial]` from the `serial_test` crate. This forces them to run one at a time across the whole binary, not just within a test case. Resolved the intermittent "EngineAlreadyExists" flakes that previously made the CI signal unreliable. Commit: `7fed112`.

## Iterative audit-fix methodology

Sparrow Engine uses a multi-round **audit-fix** cycle for final quality-gating a phase. The cycle is defined in `~/.claude/skills/audit-fix/` and has three agent roles.

| Role | Scope | Edits code | Order |
|------|-------|------------|-------|
| Auditor | Structural: simplification, dead code, deduplication, naming, readability | Yes | Parallel |
| Reviewer | Behavioral: bugs, edge cases, error handling, correctness, missing validation | Yes | Parallel |
| Inquisitor | Interrogate findings, challenge fixes, verify convergence | No (read-only) | Parallel |

### Convergence rule (absolute)

A round is CONVERGED only if all three agents find ZERO issues, propose ZERO changes, and the verification gate (tests + lint) is clean. Any round that produced findings, fixes, or concerns is NEEDS-MORE — even if the findings are re-raises from previous rounds. Only a completely clean round where nothing was found, proposed, or applied can claim convergence.

Fresh team per round. Inter-round continuity lives in the reports on disk, not in agent state. This is critical: an agent that just audited a file and approved its structure is not an independent reviewer of the resulting change.

### What the inquisitor independent scan caught that the first-round reviewers missed

Three high-value findings in this cycle:

1. **R2 — HIGH: sparrow-engine-client SRV1 wire-compat break.** R1 SRV1 extended `ModelResponse` with 5 new fields (`version`, `description`, `onnx_sha256`, `onnx_size_bytes`, `default`). The sparrow-engine-client `ModelInfo(**m)` dataclass unpacked strictly, meaning any field not in the client definition would raise `TypeError`. R1 tests used stale fixtures that didn't exercise the always-present `default: bool` field, so the test suite missed the break. R2 inquisitor's independent scan caught it. Fix: extend client dataclass with all 5 new fields (defaulted), make `load_model` symmetric, add backward-compat regression test. Commit: `4f272a8`.

2. **R3 — MI-1 LOW: stats all-NaN sentinel leak.** `summarize_detections` reset `confidence_min / max` based on `total_detections == 0`. But when detections exist with all-NaN confidences (pathological model output), the f32 fold initial values (+Inf / -Inf) leaked to output. Reset condition corrected to `non_nan_count == 0`. Regression test `all_nan_confidence_does_not_leak_sentinels` asserts 7 observable-state conditions.

3. **R3 — MI-2 MEDIUM: COCO label_id namespace collision.** When a pipeline mixes classifier-labeled rows (SpeciesNet) and detector-labeled rows (MD v6, both 1-indexed), `seen_categories.entry(label_id).or_insert_with(...)` silently drops the second `(label_id, label)` pair. Users exporting pipeline results to COCO would see wrong category names for colliding IDs — silent data corruption in the advertised `pipeline --export-format coco` path. Fix (Option A, per R4 respec): three-arm `Entry` match with first-seen semantics + `HashSet<u32>` warn dedup via `eprintln!`. R3 originally proposed `tracing::warn!`, but R4 approval-gate caught that `tracing` is not a libsparrow_engine dep and the sketch wouldn't compile.

### Why fresh team matters

The R3 auditor and reviewer (different agents than R1) both reported CONVERGED — "zero findings, zero changes." The R3 inquisitor's independent scan, starting from the code state without knowing the other agents' conclusions, found MI-1 and MI-2. Neither is trivial. Both would have shipped if the audit-fix cycle had stopped at "auditor and reviewer agree."

This validates the fresh-team-per-round rule. Reports on disk transfer what's known; agents don't inherit each other's conclusions.

### Verification gate at each round

After agents merge their worktrees to the main branch, the lead runs:

- `cargo check --workspace`
- Full test suite: `libsparrow_engine --lib`, `sparrow-engine-cli`, `sparrow-engine-server --lib`, `sparrow-engine-python --no-default-features`, `sparrow-engine-client` pytest
- Clippy: `cargo clippy --workspace --all-targets`

Any regression (previously-passing test now failing) blocks the round and is fed back to the next round as priority. Pre-existing flakes (like the engine singleton tests before `serial_test`) are distinguished from regressions by comparison against baseline.

## Fork safety testing

Python `multiprocessing` with `fork` start method breaks `ENGINE_EXISTS` in children. Not directly tested — documented limitation. Manual verification: `multiprocessing.set_start_method("spawn")` before pool creation works; default on Linux (`fork`) fails with `EngineAlreadyExists`. Expected behavior.
